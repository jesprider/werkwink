# Agent instructions

Rules for AI agents and contributors working in this repository.

---

## Identity in git artifacts

**Hard rule:** Never include your agent name, model name, or tool identity in branch
names, commit messages, PR titles, or PR descriptions.

Do not use names like `cursor`, `copilot`, `claude`, `gpt`, `agent`, or similar in
any git artifact. Use a task description; include a ticket/issue ID when one exists.

Also avoid:

- "Generated by …" or **Co-authored-by:** lines that identify an AI tool (including
  `Co-authored-by: Cursor <cursoragent@cursor.com>`)
- PR bodies that describe which agent did the work
- Branch names that encode the agent or session (e.g. `cursor/fix-bug`)
- **"Made with Cursor"** (or similar tool branding) in PR descriptions — Cursor
  may append this automatically; remove it before opening or updating a PR

**Why commits still get a Cursor trailer:** When an agent runs `git commit`, Cursor
may wrap the command and inject `--trailer "Co-authored-by: Cursor …"` via its
**Attribution** feature. That is not `AGENTS.md` being ignored and not the model
adding it to the message text. Disable in **Cursor Settings → Agent → Attribution**
(commit attribution off). This repo also ships `.githooks/prepare-commit-msg` to
strip Cursor trailers — enable once per clone: `git config core.hooksPath .githooks`.

## Branch names

Use one canonical pattern. Ticket ID is optional — include it when the work is
tracked:

```
{type}/[{TICKET-ID}-]short-description
```

| Part | Rule |
|------|------|
| `{type}` | `feature`, `fix`, `chore`, or `docs` |
| `{TICKET-ID}` | Optional. Issue or ticket ID from the tracker (e.g. `PROJ-123`, `123`) |
| `short-description` | Lowercase kebab-case, 2–5 words, describes the change |

When a ticket ID is present, it precedes the description with a hyphen. When absent,
the branch is `{type}/short-description`.

**Examples:**

```
feature/PROJ-123-add-ci-pipeline
fix/42-hill-curve-drag-offset
feature/add-ci-pipeline
docs/ci-pipeline-design
chore/bump-vitest
```

Create branches from an up-to-date `main`.

**Hard rule:** Never commit directly to `main`. Before any commit, confirm the
current branch is not `main` (`git branch --show-current`). Create and check out
a branch first. Applies to code, docs, and spec changes.

**werkwink roadmap iterations** use `{type}` = `feature` and embed the iteration
number in the slug (see `docs/superpowers/specs/2026-06-02-werkwink-v1-roadmap-design.md`):

```
feature/iteration-{N}-{slug}
```

Examples: `feature/iteration-3-persist-state`, `feature/iteration-6-store-force-mutations`.
Use `docs/…` only when the branch contains **no** implementation (spec-only PR).

Older branches named `iteration-{N}-{slug}` without a type prefix (iterations 1–5)
are legacy; do not create new ones.

## Pull requests

### Title

```
[{TICKET-ID}: ]short description
```

When the branch has a ticket ID, use the same ID in the title prefix. When there is
no ticket, use the description alone. Sentence case for the description.

**Examples:**

```
PROJ-123: Add CI pipeline
42: Fix hill curve drag offset
Add CI pipeline
Fix hill curve drag offset
```

### Description

Use this template exactly. Do not add agent identity, sign-offs, or meta commentary
about how the change was produced.

**Hard rule:** The PR body must contain **only** the Summary and Test plan sections
below — nothing before, between, or after them. Do not append footers such as
`Made with [Cursor](https://cursor.com)`; if the tooling adds one, strip it before
submitting the PR.

```markdown
## Summary

- 

## Test plan

- [ ] 
```

Fill in `Summary` with 1–3 bullets on **what changed and why**. Fill in `Test plan`
with concrete verification steps (commands run, UI paths checked, etc.).

## Commits

- **Never on `main`** — always on a typed branch (see Branch names)
- **Roman only** — no `Co-authored-by`, no `--trailer`, no tool attribution in
  commit messages
- **Never pass `--trailer`** on `git commit` (Cursor may add one anyway when
  Attribution is on; use `.githooks` or turn Attribution off)
- One logical change per commit when practical
- Message focuses on **why**, not which tool wrote it
- Follow existing repo style (imperative mood, no agent prefix)
- After committing, verify: `git log -1 --format=%B` must not contain
  `Co-authored-by: Cursor` or `cursoragent@cursor.com`

**Good:** `Add CI workflow for lint, test, and build`

**Bad:** `cursor: Add CI workflow` / `Agent: fix lint` / any `--trailer` flag

## Before opening a PR

1. Branch and PR title follow the patterns above
2. **Everything intended for the PR is committed** on the branch (see [Commits on the branch](#commits-on-the-branch) below)
3. Local checks pass:

   ```bash
   npm run lint && npm run format:check && npm run test && npm run build
   ```

4. PR description uses the template with no agent identity and **no tool footers**
   (e.g. no "Made with Cursor")

See `docs/superpowers/specs/2026-06-02-ci-pipeline-design.md` for CI details once
the workflow is merged.

### Commits on the branch

A PR must include the **full feature or fix** for that branch — not only docs or
an earlier commit while implementation stays uncommitted locally.

Before `git push` and `gh pr create`:

1. Run `git status`. If there are unstaged or uncommitted changes that belong in
   this PR, **commit them first** (unless the user explicitly asked you not to commit
   at all in this session).
2. Push the branch, then open the PR from what is on the remote branch.

### “Do not create new commits” (push / PR workflows)

Some automated or one-shot instructions say **do not stage, unstage, or create new
commits** while opening a PR. That means:

- **Do not** make extra, drive-by, or duplicate commits while running push/PR steps.
- **Do not** re-stage or rewrite history unless the user asked for that.
- **Do not** interpret this as “leave the feature uncommitted and open the PR anyway.”

If the feature code is still only in the working tree, **stop**: commit the work
(or ask the user to approve a commit), then push and open the PR.

To require the full feature in the PR, either omit the “no new commits” restriction
or say explicitly: **commit and push first, then open the PR**.
